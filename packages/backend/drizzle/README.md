# Drizzle migrations — reference only (demo phase)

The SQL files in this directory are **generated for review, not applied**.

During the MVP/demo phase the project uses `pnpm db:push` (direct schema → DB sync) for both local Postgres and Neon. The migration files exist so that schema changes show up as reviewable DDL diffs in pull requests — not because anything reads them at deploy time. There is no `db:migrate` script in the demo, and the `drizzle.__drizzle_migrations` ledger is intentionally unused.

## What lives here

- **`NNNN_<name>.sql`** — generated by `pnpm db:generate` (or `pnpm db:generate --name <descriptive>`). Each file is the cumulative diff against the previous snapshot. `0000_init_bill_pay_schema.sql` is the initial baseline.
- **`meta/_journal.json`** — Drizzle Kit's index of generated migrations. Required for `db:generate` to compute the diff for the next migration; do not delete.
- **`meta/NNNN_snapshot.json`** — full schema snapshot after each migration. Same: required for `db:generate`, do not hand-edit.

## Workflow during demo phase

1. Edit `packages/backend/src/db/app-schema.ts` (or `auth-schema.ts` via `pnpm auth:generate`).
2. Run `pnpm db:push` to sync the change to your local DB (and Neon, when deploying).
3. Run `pnpm db:generate --name <short_descriptive_name>` to commit the SQL diff alongside your schema change. Review the generated SQL — it should match what `db:push` actually did.
4. Commit both the schema edit and the migration file.

## When to graduate from this pattern

Switch to `drizzle-kit migrate` (and add a `db:migrate` script back) when any of the following becomes true:

- Real users have data on Neon you can't lose.
- More than one engineer is making schema changes (concurrent `db:push`-es will silently overwrite each other).
- You need rollback / replay against staging environments.
- CI/CD wants to run migrations as a discrete deploy step.

Graduating requires a one-time **baseline** on Neon: insert a `__drizzle_migrations` ledger row whose `hash` matches the current `meta/NNNN_snapshot.json`, so Drizzle considers the existing migrations applied. Then future migrations apply incrementally. See the [Drizzle migration docs](https://orm.drizzle.team/docs/migrations).
